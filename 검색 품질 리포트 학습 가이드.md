# `glc report` 검색 품질 리포트 학습 가이드

> 원문: `/tmp/gluck-report.md` (gluck `df680d9`, 2026-05-27 생성, fixture 30개 × 6 카테고리 사이클)
> 직전 사이클: `report-2026-05-27-3.md` (HEAD `b3c02d3`, 7개 쿼리)

---

## 한줄 요약

`glc report`는 **카테고리별로 분류된 30개의 검색 fixture**(정확 식별자/영어 자연어/한국어/오타/의역/negative)에 대해 gluck의 하이브리드 검색 엔진(BM25 + Vector + RRF)을 IR 표준 지표로 채점하는 회귀 추적 리포트입니다. 이전 사이클(7개 쿼리, MRR 0.750)보다 표본이 늘면서 **점수는 0.324로 떨어졌고 그게 정상**입니다 — 약점 영역(특히 한국어·의역)이 처음으로 측정 범위 안에 들어왔기 때문입니다.

---

## 1. 왜 이번 사이클이 분기점인가

이전 가이드는 7개 쿼리 위에서 "이제 검색 품질이 객관 지표로 추적된다"는 인프라를 다뤘습니다. 그런데 그 7개는 대부분 **영어 + 정확한 식별자/자연어** 위주여서, 측정 점수는 높아도 사용자가 실제로 던질 만한 다양한 쿼리(한국어, 오타, 의역, gluck 도메인 밖 질문)는 사각지대에 있었습니다.

이번 사이클의 핵심 변화는 다음 4가지입니다:

1. **fixture가 30개 × 6 카테고리로 확장** — 같은 5개 단위로 균형 잡혀 카테고리 간 비교 가능.
2. **카테고리별 집계** — 전체 평균만 보지 않고 어디가 무너지는지 분리.
3. **Negative query 도입** — gluck 도메인 밖 질문에 "자신 있게 틀린 답"을 주는지 검사 (Goodhart's law 방지).
4. **`aggregate`가 positive만 평균** — negative는 pass/fail이라 평균을 섞으면 의미 없음.

집계 MRR이 0.750 → 0.324로 보이지만, 직접 비교는 무의미합니다. **이전 표본은 약점을 측정하지 않는 표본이었습니다.**

## 2. 평가 파이프라인

```
search_queries.toml (30개 쿼리 + category + (expected | forbidden))
        │
        ▼
  SearchEngine.search(query, k=10)
        │
        ▼
  결과 10개
        │
        ▼
  ┌──────────────────────────────────────────────────────────┐
  │  evaluate(query, results)                                │
  │     ├─ category == Negative ──► evaluate_negative        │
  │     │     forbidden 규칙과 매칭되는 hit가 있으면 FAIL   │
  │     └─ else                  ──► evaluate_positive       │
  │           MRR / R@5 / R@10 / NDCG@10 + first_hit_rank   │
  └──────────────────────────────────────────────────────────┘
        │
        ▼
  ┌──────────────────────────────────────────────────────────┐
  │  aggregate            : positive 25개 평균              │
  │  aggregate_by_category: 카테고리별로 그룹 평균           │
  │  aggregate_negatives  : negative pass rate              │
  └──────────────────────────────────────────────────────────┘
```

### 2.1 카테고리 enum

`src/search/report/fixtures.rs::Category`

| variant | 의미 | 예시 |
|---------|------|------|
| `exact_identifier` | 식별자 정확 매칭 (BM25 강점) | `tantivy delete_term` |
| `natural_language` | 영어 자연어 | `git revwalk topological commit` |
| `korean` | 한국어 자연어 | `검색 모달 상태 머신` |
| `typo` | 영어 오타 (ngram tokenizer 내성 측정) | `tantivvy delete_trm` |
| `paraphrase` | 동의어/의역 (vector 강점이어야 함) | `합치는 점수 알고리즘` ← RRF |
| `negative` | gluck 도메인 밖 (가짜 답 안 나와야) | `react component lifecycle hooks` |

스키마 검증(loader)에서 강제하는 불변식:

- `negative` 쿼리는 `expected` 필드 금지, `forbidden` 필수.
- positive 쿼리(`negative` 외 5종)는 `forbidden` 금지, `expected` 필수.
- `forbidden` 규칙 하나는 `path` 또는 `path_prefix` 중 **정확히 하나** 지정.

이 규칙이 깨지면 `ReportError::InvalidNegativeQuery` / `InvalidForbiddenRule`가 컴파일 시점이 아닌 fixture 로드 시점에 즉시 막아줍니다.

### 2.2 기존 4가지 IR 메트릭 (MRR/Recall@k/NDCG@10)

이전 가이드의 정의와 동일하므로 요점만 다시:

- **MRR**: 첫 정답의 역순위. 1등=1.0, 미발견=0.0.
- **Recall@k**: 상위 k개에 들어온 정답 비율.
- **NDCG@10**: 상위 10개의 순위 품질을 로그 감쇠 가중치로 종합 평가.
- **Hit Rank**: 정답이 상위 10에 처음 등장한 순위. `—`는 미발견.

### 2.3 Negative query 평가: PASS / FAIL

```rust
// evaluate_negative
for each hit in top-10:
    for each rule in query.forbidden:
        if rule.path matches hit.path        → violation
        if rule.path_prefix matches hit.path → violation
passed = violations.is_empty()
```

핵심: **위반 횟수가 아니라 PASS/FAIL 이진 결과를 모으고, 그걸 pass rate로 보고합니다.** 단 한 번이라도 위반이 있으면 그 쿼리는 FAIL. 이건 "rank 9에 한 번 떴으니 90% 좋다"는 식의 변명 없이 깔끔하게 시스템을 압박하기 위함입니다.

### 2.4 좋고 나쁨의 기준선 (현재 사이클)

| 지표 | 매우 좋음 | 양호 | 개선 필요 | 현재 (positive 25) |
|------|-----------|------|-----------|-------------------|
| MRR | > 0.7 | 0.5~0.7 | < 0.3 | **0.324** |
| Recall@5 | > 0.9 | 0.7~0.9 | < 0.5 | **0.460** |
| Recall@10 | > 0.9 | 0.7~0.9 | < 0.5 | **0.600** |
| NDCG@10 | > 0.7 | 0.5~0.7 | < 0.4 | **0.399** |
| Negative pass rate | > 90% | 70~90% | < 70% | **60%** |

전체 평균이 모두 "개선 필요" 구간에 들어왔다는 건 **카테고리 일부가 0에 가깝게 깔려서 평균을 끌어내리고 있다**는 신호입니다 — 다음 섹션에서 분해해 봅니다.

### 2.5 레이턴시 (참고)

```
| p50    | p95    | p99*   | mean   | QPS    |
| 0.17ms | 0.35ms | 0.40ms | 0.20ms | 5,117  |
```

이전 사이클(p50 0.29ms, QPS 3,824)과 큰 차이는 없습니다 — 결과 품질을 보지 않는 한 검색 자체는 무시 가능한 수준의 지연. `*`는 iters=10 표본에서 p99가 표본 최댓값에 근사한다는 면책 조항.

> ⚠ 이번 리포트에는 `HEAD ≠ index.head_oid` 경고가 떴습니다. 인덱스는 `b3c02d3` 시점에 빌드되었고 현재 코드는 `df680d9`. fixture 코드와 인덱스 본문이 어긋난 채로 측정된 값이므로 다음 사이클은 `glc index --force` 후 재측정 권장. 단 한국어/paraphrase 0점의 근본 원인은 인덱스 stale이 아니라 토크나이저·임베딩 한계입니다(아래 Q1 참고).

## 3. 핵심 발견: By Category 해부

```
| Category         | n | MRR   | R@5   | R@10  | NDCG@10 |
| exact_identifier | 5 | 0.620 | 0.800 | 1.000 | 0.710   |  ← 강함
| natural_language | 5 | 0.600 | 0.900 | 1.000 | 0.658   |  ← 강함
| korean           | 5 | 0.000 | 0.000 | 0.000 | 0.000   |  ← 완전 무력
| typo             | 5 | 0.362 | 0.400 | 0.800 | 0.550   |  ← 부분 내성
| paraphrase       | 5 | 0.040 | 0.200 | 0.200 | 0.077   |  ← 거의 무력
```

### 3.1 강한 영역 (exact_identifier, natural_language)

- **R@10 = 1.000** — 5개 모두 정답을 상위 10에 잡음. "찾기"는 완성.
- MRR 0.6x → 정답이 평균 1~2등 사이. 직전 사이클(이 영역만 보면 MRR 0.75)에서 약간 흔들렸지만 한 fixture(`build_index_incremental` 단독 검색이 rank 10에 잠김)가 평균을 끌어내린 것이고 본질적 후퇴는 아님.
- 결론: **이 영역은 더 이상 흥미로운 게임이 아닙니다.** rank 1 vs rank 2 사이의 미세 튜닝은 사용자 체감 차이가 거의 없음.

### 3.2 한국어 — R@10 = 0.000

5개 쿼리 전부 상위 10 안에 정답이 한 건도 안 들어왔습니다. 검색이 "조금 잡는데 순위가 낮은" 게 아니라 **아예 못 잡는** 상태.

원인 후보:

1. **BM25 `ngram_2_2` 토크나이저**: 영어 식별자에는 잘 맞지만 한국어는 음절·자모 경계가 다름. 예: "검색 인덱스 빌드"에서 ngram이 한글 바이트를 잘라 의미 없는 토큰을 만들 가능성.
2. **임베딩 `potion-multilingual-128M`**: 다국어 학습이지만 한국어 코드 컨텍스트 문서가 학습 데이터에 거의 없었을 가능성. 256-dim 경량 모델이라 표현력 한계도 있음.
3. **인덱싱 본문이 한국어를 포함하지 않음**: src/ 코드와 커밋 메시지는 영어 중심. "검색 인덱스 빌드"에 의미적으로 가까운 `src/search/indexer.rs` 본문에는 한국어가 없음. 벡터 검색이 다국어 임베딩으로 영-한 간 의미 유사도를 잡아줘야 하는데 실패.

이 단계에서 곧바로 해결책으로 점프하지 말고, **다음 사이클에서 BM25/Vector 각 단계의 raw 점수를 로깅**해서 어디서 끊기는지부터 확인해야 합니다. 인덱싱 단계의 토크나이저 출력만 봐도 (1)인지 (2)~(3)인지 구분 가능.

### 3.3 paraphrase — MRR 0.040

5개 중 1개만(`합치는 점수 알고리즘` → `rrf.rs` rank 5) 부분적으로 잡고 나머지 4개는 0점. paraphrase는 본래 **vector 검색이 끌어올려야 하는** 카테고리입니다 — "벡터 압축 방식"은 본문 어딘가에 "quantization", "4bit" 같은 단어가 나오는 `vector.rs`를 잡아야 하는데 못 잡음. 한국어 paraphrase가 4개라는 점에서 한국어 약점과 원인이 겹칠 가능성 큼.

`합치는 점수 알고리즘`만 잡힌 이유 추정: "합치는"이 영어 "fusion"의 의역인데, `rrf.rs` 본문에 "fusion"이 명시되어 있고 RRF 라는 줄임말도 path에 등장 → 다국어 임베딩이 어느 정도 작동하는 케이스. 즉 paraphrase가 일관되게 안 되는 게 아니라 **본문에 의역 단서가 풍부한 파일만 살아남고 있다**는 신호.

### 3.4 typo — MRR 0.362

ngram_2_2의 효과가 부분적으로 보이는 영역:

- `embeding model potion` → rank 1 (1글자 결손도 ngram이 흡수)
- `tantivvy delete_trm` → rank 2 (2글자 변형도 흡수)
- `build_index_incremntal`, `reciprical rank fussion` → rank 6~7 (잡지만 깊이 떨어짐)
- `revwalk topolgical` → 미발견 (`store.rs` 본문에 "revwalk"가 명시적 토큰으로 안 나타날 가능성)

ngram tokenizer는 **잡긴 잡지만 순위를 끌어올리진 못합니다**. 즉 R@10은 0.8로 양호하지만 MRR이 낮음. 트레이드오프 정상 동작.

## 4. Negative Queries — Goodhart's law 방지 장치

```
n=5, pass rate 60%
| 1 | react component lifecycle hooks       | PASS
| 2 | django orm migrations                 | FAIL (rank 6/8/9: src/...)
| 3 | kubernetes pod scheduling             | PASS
| 4 | spring boot dependency injection      | FAIL (rank 7: src/search/report/render.rs)
| 5 | tensorflow model training loop        | PASS
```

### 4.1 왜 필요한가

긍정 메트릭(MRR/NDCG)만 보고 검색을 튜닝하면, "어떤 쿼리에 대해서든 그럴듯한 답을 내놓는" 시스템이 메트릭상 유리해집니다. 사용자가 코드베이스와 관계없는 질문을 던졌을 때 자신 있게 `src/main.rs`를 1위로 추천하는 시스템은 메트릭 점수와 무관하게 신뢰 못 할 도구. **Negative query는 이 시스템이 "모른다"고 말할 수 있는지를 측정합니다.**

### 4.2 이번 사이클이 알려준 것

- **PASS 3개** (react, kubernetes, tensorflow): 검색이 src/ 하위 결과를 상위 10 밖으로 밀어냄. 정답.
- **FAIL: django orm migrations** — rank 6, 8, 9 세 자리에 src/ 결과 노출. "migrations"라는 단어가 `indexer.rs`("indexing")와 부분 매칭됐을 가능성. 3건 위반이라 가장 약함.
- **FAIL: spring boot dependency injection** — rank 7에 `src/search/report/render.rs`. 의외의 매치. 본문에 "report"/"injection" 같은 우연한 토큰 겹침일 가능성.

pass rate 60%는 "개선 필요" 영역(< 70%). 다만 두 FAIL 모두 **rank 6 이하**에 위반이 모여 있다는 점에서 사용자 체감으로는 덜 치명적입니다(상위 5에는 안 나옴). 이걸 메트릭화하려면 *"violation at rank ≤ 5"* 같은 더 엄격한 변형을 도입할 수 있지만 현재 단계에선 이진 PASS/FAIL이 충분.

### 4.3 한 가지 미묘함

`negative` 카테고리는 `aggregate` 평균에서 의도적으로 **제외**되어 있습니다. MRR/Recall은 정답이 있는 쿼리에서만 의미가 있고, negative와 섞으면 평균이 왜곡됩니다. 그래서 리포트의 `## Aggregate`에 `(positive only, n=25)`라고 명시되어 있고, negative는 별도 `pass rate` 한 줄로 나옵니다. 두 지표를 동시에 봐야 정직한 평가가 됩니다.

## 5. 이해도 확인 Q&A

### Q1. MRR이 0.750에서 0.324로 떨어졌습니다. 검색이 후퇴한 건가요?

**A**: 아니요. **표본을 다른 것으로 바꿨기 때문**입니다. 이전 7개 쿼리는 영어 + 정확 식별자/자연어 비중이 높았고, 그건 정확히 BM25 + path_terms가 강한 영역이었습니다. 이번 25개 positive에는 한국어 5개(전부 0점), paraphrase 5개(거의 0점)가 새로 들어와 평균을 끌어내렸습니다. 강한 영역만 비교하면 이번 사이클의 `exact_identifier` 0.620 + `natural_language` 0.600은 직전 0.750과 거의 같은 레벨(약간의 단일 쿼리 노이즈 범위 안). **메트릭 변화의 원인을 카테고리별로 분해하지 않고 전체 평균으로만 보는 건 위험합니다.**

### Q2. 한국어 R@10이 0인 게 인덱스 갱신 안 됨 때문일 수도 있나요?

**A**: 아니, 거의 아닙니다. `glc report`의 HEAD mismatch 경고는 "현재 코드가 인덱스 빌드 당시와 다르다"는 뜻이지만, 이번 변경은 `src/search/report/...` 와 fixture 파일에 국한되었고 **인덱싱 대상 파일들의 본문은 거의 그대로**입니다(`src/search/indexer.rs`, `embedding.rs` 등). 한국어 쿼리가 0점인 본질은 (1) BM25 ngram이 한국어 음절을 의미 토큰으로 못 자르거나, (2) 다국어 임베딩이 한국어 쿼리 ↔ 영어 코드 컨텍스트의 의미 유사도를 충분히 잡지 못하거나, (3) 인덱스 본문에 한국어가 거의 없어서 매칭할 표면이 없는 것. 검증은 인덱스를 새로 빌드한 뒤 BM25/Vector raw 점수를 로깅해서 어디서 끊기는지 보는 게 정답입니다.

### Q3. paraphrase 5개 중 1개만 잡혔습니다. 왜 그것만 잡혔나요?

**A**: 잡힌 쿼리는 `합치는 점수 알고리즘` → `src/search/rrf.rs` (rank 5). 단서: RRF 본문에 "fusion"이라는 영어 단어가 명시적으로 들어가 있고 path에 "rrf"라는 토큰이 있어서 **다국어 임베딩이 "합치는" ↔ "fusion"의 의미적 유사도를 어느 정도 잡아주기에 충분한 표면**이 있었습니다. 나머지 4개는 본문에 대응 영어 토큰이 충분히 강하지 않거나 표면적이 좁은 파일(`chunk/mod.rs`, `mod.rs`)이 정답이라 신호가 약합니다. 결론: 임베딩이 작동을 안 하는 게 아니라, **한국어 쿼리 → 영어 본문 매칭은 본문 어휘에 영어 단서가 풍부할 때만 살아남는 임계 구조**입니다.

### Q4. Negative pass rate 60%는 어떻게 해석하나요?

**A**: 5개 중 3개는 PASS, 2개는 FAIL. 두 FAIL 케이스 모두 위반이 rank 6 이상(상위 5 밖)에 있어서 사용자가 화면 첫 페이지에서 잘못된 답을 볼 가능성은 낮습니다. 그렇다고 안심할 일은 아닌데, 검색의 신뢰성은 "위에서 보면 괜찮음"이 아니라 "끝까지 일관됨"이기 때문입니다. 다음 사이클에서는 (1) 위반 케이스의 *rank 분포*를 메트릭에 반영하거나, (2) "violation at top-5"라는 더 엄격한 변형을 추가 보고 항목으로 넣는 게 자연스러운 발전 방향입니다. 단, 이걸 지금 점수에 끼워 넣으면 알람이 너무 자주 울리니까 기준선부터 잡은 다음 단계적으로 강화.

### Q5. `aggregate`가 negative를 제외하는 이유는 뭔가요?

**A**: 두 종류의 메트릭이 의미하는 게 다르기 때문입니다. positive 쿼리의 MRR/Recall/NDCG는 "정답이 어디 있는지"를 측정하고, negative 쿼리의 결과는 "있어선 안 될 게 있는지"라는 이진 판정입니다. 둘을 한 평균에 섞으면 (a) negative 쿼리의 MRR을 강제로 0으로 매기든(잘 통과한 케이스도 0점 처리) (b) 1로 매기든(위반 케이스도 만점 처리) 어느 쪽이든 의미 없는 숫자가 나옵니다. 그래서 코드(`aggregate(per_query)`)는 명시적으로 `QueryEval::Positive`만 필터링하고, negative는 별도 `aggregate_negatives`로 pass rate를 따로 보고합니다. 리포트의 `## Aggregate` 헤더가 `(positive only, n=25)`라고 표시하는 이유.

### Q6. 다음 임베딩 모델을 더 큰 걸로 바꾸면 한국어가 살아날까요?

**A**: 아마 일부 살아납니다. 하지만 **그게 옳은 첫 수는 아닙니다.** 이유: (1) 모델 교체는 인덱스 크기·빌드 시간·로컬 메모리 사용량을 동시에 끌어올리는 비싼 변경, (2) 한국어 0점의 원인 중 BM25/벡터 어느 쪽이 더 크게 기여하는지 아직 데이터로 확인 안 됨, (3) 임베딩만 바꿔도 BM25 ngram이 한국어 음절을 못 토크나이즈한다면 BM25 부분은 여전히 0, (4) 측정 가능한 자리에서 임베딩 A/B를 비교하려면 우선 fixture 카테고리별 raw 점수 로깅이 먼저 들어와야 함. 권장 순서: **먼저 진단 → 그다음에야 모델 교체**.

### Q7. fixture 정답이 잘못 적혀 있을 가능성은 어떻게 잡나요?

**A**: 두 가지 안전망이 있습니다. (1) loader 단계의 스키마 검증 — `negative`에 `expected`를 같이 적거나, `forbidden` 규칙에 `path`와 `path_prefix`를 동시 지정하면 즉시 거부. (2) **R@10이 갑자기 1.0에서 0.0으로 점프하면 fixture/코드 정합성 의심**. 이전 사이클에서 `modal_state.rs`로 파일이 리네이밍되면서 fixture가 자동으로 살아난 사건이 좋은 예. 코드 리네이밍/파일 이동을 한 사이클에서는 fixture path를 grep으로 한 번 훑는 게 안전합니다.

### Q8. 다음 사이클의 우선 작업은 뭔가요?

**A**: 우선순위 정렬은 **"점수 끌어올리기"보다 "왜 0점인지 가시화"** 가 먼저입니다.
1. **HEAD == index.head_oid 보장** 인덱스 갱신 후 재측정 — 기준선 자체를 깨끗하게.
2. **BM25 raw 토큰 로깅** 한국어 쿼리를 ngram이 어떻게 자르는지 확인. 한 줄짜리 디버그 출력이면 충분.
3. **Vector raw 점수 로깅** 같은 한국어 쿼리에서 벡터 검색이 0점인지 단순히 BM25에 밀리는 건지 분리.
4. **카테고리별 임계값 설정** 예: korean R@10 > 0.3, paraphrase MRR > 0.2 같은 다음 목표를 명시. 전체 평균 하나로 통과/실패를 보면 안 됨.
5. **임베딩 모델 후보 비교는 1~4가 끝난 뒤**. 그땐 fixture 30개 위에서 카테고리별로 차이가 정량 비교됨.

피해야 할 함정:
- **negative 쿼리에 맞추려고 코드 본문을 수정** (= Goodhart's law). 본문 수정 없이 토크나이저/임베딩만 건드릴 것.
- **fixture를 더 늘려서 평균을 흐리기**. 30개 카테고리 구조 안에서 정확도부터 끌어올리는 게 우선.

## 6. 더 알아보기

### 핵심 키워드
- **Goodhart's law**: "측정 가능한 것이 목표가 되면 그건 더 이상 좋은 측정값이 아니다." Negative query 도입 동기.
- **Cranfield paradigm**: 정답 라벨링된 쿼리 세트로 검색을 평가하는 고전적 방법론.
- **TREC Robust track**: 어려운 쿼리(부정·드문 토픽 포함)를 의도적으로 섞어 평가하는 NIST 벤치마크 시리즈.
- **Stratified evaluation**: 카테고리별 분리 평가. 전체 평균이 약한 영역을 가리는 걸 막음.

### 관련 코드 (이번 사이클 변경 지점)
- `src/search/report/fixtures.rs`: `Category` enum, `ForbiddenRule`, 스키마 검증 로더.
- `src/search/report/metrics.rs`: `QueryEval::{Positive, Negative}`, `evaluate_negative`, `aggregate_by_category`, `aggregate_negatives`.
- `src/search/report/mod.rs`: `Report.by_category`, `Report.negatives` 필드.
- `src/search/report/render.rs`: `## By Category`, `## Negative Queries`, Per-Query `Cat` 컬럼.
- `tests/fixtures/search_queries.toml`: 30개 쿼리 × 6 카테고리.

### 진단 도구로 가는 다음 단계 워크플로

```
1. main에서 `cargo run --bin glc -- index --force`
2. `glc report --out baseline.md` → 이번 카테고리 메트릭을 baseline으로 고정
3. 검색 변경 (BM25 토크나이저 / 임베딩 모델 / RRF 가중치)
4. 같은 fixture로 `glc report --out candidate.md`
5. **카테고리별 diff를 계산해서 어느 영역이 어떻게 움직였는지** 확인
6. 한 카테고리가 좋아진 대신 다른 카테고리가 무너지지 않았는지 검사
7. negative pass rate가 하락하지 않았는지 검사
```

평균 한 줄로 회귀를 판단하면 안 되는 이유가 이번 사이클의 가장 큰 학습입니다.
