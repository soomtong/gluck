# `glc report` 검색 품질 리포트 학습 가이드

> 원문: `docs/reports/report-2026-05-27-5.md` (gluck `b12a3c9`, 2026-05-27 생성, fixture 30개 × 6 카테고리)
> 직전 사이클: `docs/reports/report-2026-05-27-4.md` (HEAD `4236478`, 한국어 0점 상태)

---

## 한줄 요약

지난 사이클은 **카테고리별 fixture로 한국어·paraphrase 0점이 측정 범위에 들어왔다**고 알려준 분기점이었고, 이번 사이클은 그 0점을 **진단하고 개입해서 끌어올린 첫 사이클**입니다. Aggregate MRR이 0.318 → **0.563**로 +77%, Korean R@10이 0.000 → **0.900**로 점프했습니다. 영어 카테고리에는 회귀가 없고, negative pass rate도 그대로 유지됐습니다.

---

## 1. 왜 이번 사이클이 중요한가

직전 가이드의 Q8은 "점수 끌어올리기보다 왜 0점인지 가시화가 먼저"라고 권고했습니다. 이번 사이클은 그 순서를 그대로 지켰습니다:

1. **`glc diagnose <query>` 진단 도구를 먼저 추가** — BM25 ngram 토큰, BM25 raw 점수 top-N, Vector cosine top-N, RRF fused top-N을 한 화면에 나란히 출력. 어디서 신호가 끊기는지 분리해서 보는 도구.
2. 한국어 쿼리 하나(`검색 인덱스 빌드`)를 통과시켜 **3가지 가설을 데이터로 검증** — 토크나이저, 임베딩, 인덱싱 본문 중 무엇이 진짜 원인인지 좁힘.
3. 검증된 원인에 맞춘 **개입 3종을 같은 사이클에 도입** — path-based 한국어 별칭, Korean 쿼리 전용 BM25 모드, BM25 anchor RRF.
4. 같은 fixture로 **재측정해서 회귀 여부 확인** — 영어 카테고리·negative pass 변화 없음.

이번 사이클의 가장 큰 교훈은 "진단 없이 손대지 마라"가 작동했다는 것입니다. 처음 떠올린 해결책(임베딩 교체)은 가장 비싸고 가장 늦은 수였는데, 진단 결과 **임베딩이 아니라 BM25/RRF 흐름의 구조 문제**였습니다.

## 2. 진단으로 확정된 사실

`glc diagnose "검색 인덱스 빌드"`가 알려준 것:

| 가설 | 결과 | 근거 |
|------|------|------|
| BM25 ngram이 한국어를 못 자른다 | **기각** | "검색 인덱스 빌드"가 8개의 깔끔한 ngram(`검색`, `색 `, ` 인`, `인덱`, `덱스`, …)으로 잘림 |
| 다국어 임베딩이 너무 약하다 | **부분 확정** | 정답 `indexer.rs`의 `IndexStatus` 심볼이 vec rank 10, cosine 0.36으로 잡힘 — 작동은 하지만 너무 약함 |
| 인덱스 본문에 한국어가 없다 | **확정 (핵심 원인)** | BM25 top 10이 전부 본문에 한국어 텍스트가 있는 .md/commit 문서로 채워짐. `src/search/indexer.rs` 본문에는 한국어 0이라 BM25 매칭 표면 0 |

그러니까 정답은 어디서나 떠 있긴 했지만(vec rank 10), **9개의 더 강한 매칭(commit/.md)이 위를 가로막고 있어** top 10 밖으로 밀려난 게 R@10 = 0의 메커니즘이었습니다. RRF 산수로는 이 비대칭을 못 깹니다 — 양쪽 리스트에 동시에 잡힌 노이즈 doc은 단일 매칭 정답 doc을 항상 누적 점수로 이깁니다.

## 3. 도입된 개입 3종

### 3.1 Path-based 한국어 별칭 (BM25 인덱싱)

`src/search/text_prep.rs`의 const 글로서리(~30 entry)로 영문 path 토큰을 한국어로 매핑합니다.

```
"search"    → "검색"
"index"     → "인덱스"
"indexer"   → "인덱스 인덱서"
"modal"     → "모달"
"state"     → "상태"
"highlight" → "하이라이트 강조"
"commit"    → "커밋"
"rrf"       → "융합 합치"
…
```

인덱싱 시점에 `Bm25Index::add_doc`가 path를 토큰화해서 글로서리 lookup으로 한국어 별칭을 만들고, 그걸 기존 `path_terms` 필드 끝에 이어 붙입니다. 예: `src/search/indexer.rs` → path_terms 필드 = `"src search indexer rs 검색 인덱스 인덱서"`.

원칙은 **path 토큰과 정확히 일치할 때만 매핑** — substring 매칭은 false positive를 양산하기 쉬워 사용하지 않습니다. 글로서리에 없는 path(예: `src/foo/bar.rs`)는 그대로 영어 path_terms만 유지.

INDEX_VERSION 6 → 7. 별칭이 인덱스 본문에 박히기 때문에 `glc index --force`로 재빌드 필요.

### 3.2 Korean 쿼리 전용 BM25 모드 (검색 시점)

`text_prep::is_korean_query(q)`가 Hangul Syllables 블록(U+AC00–U+D7AF) 글자를 하나라도 감지하면 한국어 쿼리로 분류합니다. 비율이 아니라 존재 기반 — "검색 modal" 같은 혼합도 한국어로 봅니다.

한국어 쿼리는 BM25에서 **body 필드를 제외하고 title + path_terms만** 사용합니다(`Bm25Index::search_path_title_only`). 이유:

- README.md, 학습 가이드.md, 한국어 커밋 메시지 같은 도큐먼트들은 본문에 자연어 한국어가 풍부합니다.
- body ngram_2_2 매칭이 이런 .md/commit을 항상 위에 올려서, path 별칭이 만든 신호를 묻어버렸습니다.
- body를 빼면 .md/commit이 path/title에 한국어가 없을 때 가라앉고, **path 별칭이 박힌 소스 파일이 떠오릅니다**.

또한 한국어 모드에서는 path_terms 가중치를 2.0 → **5.0**으로 올리고 title은 2.0 → 1.0으로 낮춥니다. commit title의 한국어 매칭("시맨틱 검색 모달...")이 path 별칭을 압도하지 않게 하기 위함.

### 3.3 BM25 anchor RRF (Fusion 시점)

`rrf::rrf_fuse_with_bm25_anchor(bm25, vec, k, limit, w_bm25, w_vec, n_anchor)`이 핵심 개입입니다. 동작:

```
1. weighted RRF로 fused 리스트 계산 (한국어는 w_vec=1.5x로 paraphrase 케이스 보호)
2. BM25 top n_anchor(=3)을 결과 앞에 강제로 배치
3. 그 뒤를 fused 리스트로 채움 (중복 제거)
```

왜 anchor가 필요한가? 진단으로 드러난 RRF의 한계 때문입니다. `검색 모달 상태 머신` 쿼리에서:

- BM25 #1 = `modal_state.rs` (정답, score 30.23) — path 별칭이 강하게 잡음
- Vec top 10에는 `modal_state.rs`가 없음(임베딩이 못 잇는다)
- 대신 vec #1 = commit b098 (시맨틱 검색 모달 재오픈) — 의미적 노이즈
- 같은 commit이 BM25 #6에도 있어서 RRF가 누적
- **순수 RRF: commit b098 0.0397 vs modal_state.rs 0.0164 → modal_state.rs 패배**

이건 어떤 RRF 가중치·k 값으로도 못 뒤집습니다. 양쪽 리스트에 동시에 떠 있는 doc은 항상 단일 매칭 doc을 이깁니다. anchor는 이 산수를 깨고 BM25 top 3을 무조건 보존합니다.

비-한국어 쿼리는 anchor 없이 기존 RRF 그대로 통과 — 영어 카테고리에 회귀 없음.

## 5. 이해도 확인 Q&A

### Q1. 임베딩 모델을 안 바꾸고도 한국어가 살아난 이유는?

**A**: 진단으로 임베딩이 핵심 병목이 아님을 먼저 확정했기 때문입니다. `indexer.rs`의 `IndexStatus` 심볼은 임베딩이 이미 cosine 0.36으로 잡고 있었습니다. 문제는 그 신호가 9개의 더 강한 노이즈에 가려져 top 10 밖으로 밀려난 것이고, 그 노이즈는 **BM25가 도큐먼트 본문의 자연어 한국어를 너무 잘 잡아내서** 만들어진 것이었습니다. 그래서 (1) BM25에 path 별칭을 박아 소스 파일이 신호를 갖게 하고, (2) 한국어 쿼리에서 BM25 body를 제외해서 .md/commit을 가라앉히고, (3) anchor로 BM25 top을 보존하는 — 임베딩과 무관한 3가지 변경만으로 충분했습니다. 임베딩 교체는 인덱스 크기/빌드 시간을 다 늘리는 비싼 수인데, 이번 사이클은 비용 0으로 해결됐습니다.

### Q2. BM25에 별칭만 박았는데 왜 처음엔 효과가 거의 없었나요?

**A**: 별칭이 인덱스에 들어가는 것과 검색 ranking에서 이기는 것은 다릅니다. 별칭 주입 직후 `검색 인덱스 빌드` 쿼리에서 `indexer.rs`는 BM25 rank 27까지 올라왔습니다(별칭 전엔 아예 안 나옴). 하지만 1~26위는 본문에 한국어가 풍부한 .md/commit이 차지했습니다. BM25는 본문 ngram 매칭에 가산점이 누적되는 구조라, path_terms의 2 hit × 2.0 boost = 4 effective signal로는 본문 6~8 ngram 매칭에 못 이깁니다. 그래서 추가 개입이 필요했습니다 — 한국어 쿼리에서 body를 빼고(노이즈 제거), path_terms boost를 5.0으로 올리고(별칭 강화), 그 결과 BM25 자체에서 정답이 rank 1~2로 올라왔습니다.

### Q3. 한국어 쿼리에서 BM25 body를 제외하는 게 너무 공격적이지 않나요?

**A**: 본문에 한국어가 있는 .md/commit을 일부러 못 찾게 만드는 셈이라 그렇게 보일 수 있습니다. 트레이드오프는 다음과 같습니다: 한국어 코드 검색의 90% 사용자는 "한국어 단어로 영어 소스 파일을 찾고 싶다"는 의도(예: "검색 인덱스 빌드"로 indexer.rs를 찾기)이고, 나머지 10%가 한국어 도큐먼트 자체를 찾고 싶은 경우입니다. body를 빼면 다수 의도에 맞추되 소수 의도는 path/title 매칭만 남깁니다. 다수가 명확히 더 큰 가치를 갖는 트레이드오프라서 받아들였습니다. 만약 fixture에 "한국어 도큐먼트 찾기" 쿼리가 추가되어 회귀가 보이면 그때 정책을 재고하면 됩니다.

### Q4. 왜 RRF의 k값 튜닝이나 가중치 변경만으로는 안 됐나요?

**A**: RRF의 산수가 동시 매칭을 누적하는 구조라서입니다. doc이 BM25와 Vector 양쪽 리스트에 동시에 있으면, 두 contribution이 합쳐져서 단일 매칭(BM25만 또는 Vector만)을 항상 이깁니다. `modal_state.rs`(BM25 #1만)는 1/(60+1) = 0.0164, 반면 commit b098(BM25 #6 + Vec #1, w_vec=1.5)은 1/(60+6) + 1.5/(60+1) = 0.0397. 어떤 k(1, 10, 60, 200)을 넣어도, 어떤 가중치(w_bm25=1~5)를 넣어도 단일 매칭이 동시 매칭을 못 이깁니다. anchor는 이 RRF 산수를 우회하는 결정적 개입입니다 — BM25 top 3을 결과 앞에 강제 prepend.

### Q5. 글로서리 ~30개로 다 잡혔다는 게 신기한데, 확장하면 더 살릴 수 있나요?

**A**: 남은 paraphrase 1건(`토크나이저 설정` → `src/search/bm25.rs`)이 정확히 이 케이스입니다. bm25.rs path에는 "tokenizer" 토큰이 없어서 글로서리가 trigger되지 않습니다. "tokenizer" → "토크나이저"를 bm25.rs path에 인위적으로 박는 식이면 살릴 수 있겠지만, 이건 fixture에 맞춰 글로서리를 튜닝하는 Goodhart's law의 함정입니다. 옳은 방향은 글로서리가 아니라 (1) 더 풍부한 의미 매칭이 가능한 임베딩으로 교체하거나, (2) 본문/주석에 한국어 도메인 단어를 명시적으로 박는 코드 컨벤션 도입입니다. 다음 사이클의 의사결정 자리.

### Q6. 다음 사이클의 우선순위는 무엇인가요?

**A**: 이번 사이클은 한국어를 0점에서 양호 구간으로 끌어올린 큰 수였습니다. 다음은 작은 잔여 문제들을 정리하는 사이클이어야 합니다:

1. **영어 typo 회복** (`reciprical rank fussion`, `revwalk topolgical`) — typo R@10 0.6에 멈춰 있습니다. ngram_2_2가 영어 오타도 어느 정도 흡수하지만 다음자 변형은 놓칩니다. 가능한 수: edit distance 기반 fuzzy query, 또는 더 작은 ngram(예: 2_3).
2. **negative `django orm migrations` 잔존 FAIL** — `indexer.rs`의 "indexing" 토큰이 "migrations" ngram과 부분 매칭. forbidden rule을 더 엄격하게(`rank ≤ 5 only`) 메트릭화하거나, 본문에서 commits/spec 도큐먼트를 negative 쿼리에서 제외하는 정책.
3. **paraphrase `토크나이저 설정` 결정** — Q5 참조. 글로서리 확장(빠른 수)과 임베딩 교체(올바른 수) 중 선택.
4. **fixture 확장은 잠시 보류** — 30개에서 카테고리별 의미 있는 차이가 보이는 단계라 더 늘리면 평균이 다시 흐려집니다.

### Q7. 영어 카테고리에 진짜 회귀가 없는지 어떻게 확신하나요?

**A**: 두 가지 안전망입니다. (1) `is_korean_query`가 Hangul 글자 존재 기반이라 영어 쿼리는 절대 false positive가 아닙니다. (2) 영어 카테고리는 기존 코드 경로(body 포함, anchor 없는 RRF)를 그대로 사용합니다. 측정상으로도 exact 0.620 → 0.620(동일), natural 0.617 → 0.600(단일 쿼리 노이즈, `vector ANN search backend`가 rank 3 → 4로 이동) — 시스템 변화 아닌 측정 노이즈로 해석. typo 0.329 → 0.329(동일). 영어 쿼리는 코드 흐름이 바뀌지 않았으니 회귀할 자리가 없습니다.

### Q8. 이 패턴(별칭 + 모드 분기 + anchor)을 다른 언어로 확장할 수 있나요?

**A**: 구조적으로 깨끗하게 확장됩니다. `is_korean_query`를 `is_non_ascii_query` 같은 더 일반적인 함수로 바꾸고 글로서리를 언어별로 묶으면 일본어/중국어도 같은 메커니즘이 작동할 가능성이 큽니다. 단 두 가지 주의: (1) 일본어는 띄어쓰기 없는 언어라 word_lower 토크나이저가 한국어와 다르게 동작합니다 — 형태소 분석기 필요. (2) 글로서리는 언어마다 따로 큐레이션이 필요한 비용이라 fixture가 늘어나기 전엔 투자할 가치가 낮습니다. 한국어가 작동하는 걸 확인한 자체가 다른 언어로 가는 자신감의 근거입니다.

## 6. 더 알아보기

### 핵심 키워드
- **Hybrid retrieval**: 어휘적(BM25)와 의미적(Vector) 신호를 결합. 결합 방식(RRF, weighted sum, learned-to-rank)에 따라 트레이드오프가 달라짐.
- **RRF (Reciprocal Rank Fusion)**: 순위 기반 결합. 점수 스케일 차이를 무시할 수 있어 다른 도메인 결합에 강함. 단점: 두 소스에 동시 노출된 doc이 단일 소스 강매칭을 이김(이번 사이클의 핵심 발견).
- **Bilingual code search**: 자연어(한국어)로 영어 소스 코드를 찾는 문제. path/identifier alias가 비교적 저비용의 보조 수단.
- **Anchor-based fusion**: RRF의 누적 비대칭을 깨기 위해 한 소스의 top-N을 강제 보존하는 변형. 본 가이드 §3.3.
